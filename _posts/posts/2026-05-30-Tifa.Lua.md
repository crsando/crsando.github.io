最近又开始倒腾的 ***tifa.lua** （量化交易框架）了

最近主要考虑这么几个事情

### CTP Query 改队列

最早实现的时候，我的设想是用 *lservice* 框架，单独跑一个 *ctp_trader.lua*，这个里头所有的交易（*order*）和各种查询（*query*）都是并发的，把回调函数通过消息队列改成同步的语义。

这个设计本身还是ok的，但是当时留了一个坑：等我把代码写好，我才发现，*CTP*服务器是有流控的，如果一次同时发送多个 *Query* 会被限制。

之前用了一个非常恶心的实现来保证一次只处理一个 *query* ：设置只一个 *flag*，有*query*就把这个*flag*设置为开，其他 *query* 看到这个 *flag* 就先暂定。

这次索性直接把他改成了队列的形式，逻辑就是，比如 *A* 服务 *call* 了一下 *ctp_trader.lua* 的 *query_account*，那么*query_account* 并不会直接执行，而是先把这个 *query* 丢入一个队列。队列里的 *query* 是一次执行一个的，执行完的结果再返回 *A* 服务。

### lservice2 -> lservice3

之前这个项目的底层框架基础是我自己开发的 ***lservice2***，这个项目主要借鉴的是 *skynet/ltask* 的模型，但是大幅简化了。现在的大概逻辑是：

- 每个 *lua_State* 跑在一个独立的线程中，作为一个 ***service***
- 不同线程之间只能通过消息来传输数据
- 表现在方法上，体现为可以互相*call*，比如 `service.call(user, "query_account")` 这样
- 每个 *call* 是 *blocking*的。 对应在 *user.lua* 中，我们实现一个 `function S.query_account()` 来接住这个*call*
- 这个函数在执行中实际上会被封装成一个 *coroutine*，也就是一个单独的执行绪，每个 *corotine* 都可能被挂起（比如*call*了另一个服务），来实现同步非阻塞的一种效果。每个*coroutine*内部可以理解成是顺序执行的。

这里面，不同服务之间的消息传递有两个基础：一个无锁的消息队列存消息以及相应的*recv/send*，一个 *conditional variable* 用来通知对应线程（得到了消息，需要*recv*）


结合*CTP*，我们把相关的回调函数，比如`OnRspQueryAccount`，借用这个消息队列，把回调改成同步的模式，比如用伪代码大概是这么一个效果

```lua
function S.query_account()
    local req_id = trader:query_account() -- 发消息
    local ok, rst = coroutine.yield()
    return rst
end
```

也就是我们会主动挂起当前执行的*coroutine*，等待`OnRspQueryAccount()`返回结果后，从另外一个消息处理的函数中，去把这个 *coroutine.resume*

这个模式本身没有什么问题，但是对我来说的最大挑战是：他没有办法利用一些现成的异步库，我还需要结合这个模型自己写一套 *socket*, *timer* 之类的东西。

说实话，本身 *skynet/ltask* 我研究的不够深入，没有弄明白他里头的对应实现机制（但是应该是自己实现了一套*socket*）。

我最近仔细想了一想，还是想回到之前就有过的一个想法：能否把现成的 ***libuv*** 利用起来？

最近借住*AI*的帮助，感觉自己研究的又深入了一点了，发现这个想法是可行的，而且还挺方便。

所以在原本的基础上改造了 *lservice3*，改动的地方在于

- 原本每个*service*比如 `user.lua` 这种，就是跑在新的 *thread*中，所以每个 `user.lua` 最后其实是一个 `while true do`，其中用阻塞式的 `recv` 来读取跨现成通信的消息。
- 现在我们每个 *service* 其实都有单独的一个 *libuv*的 *loop*，所以最后一步实际上是在 *C-side* 的 `uv_run`
- 跨服务通知不再使用 *conditional_variable*，而是直接使用 *uv_async_t / uv_async_send*，我们在 *uv_async_cb* 这个回调中执行消息处理。消息队列不用改，直接复用。
- 由于每个服务都有独立的 *loop*，所以我们可以直接在这个 *loop* 上跑 *luv*（*libuv*的*lua-binding*）提供的各种 *socket*, *timer* 类的功能，不需要自己重新写代码

整个改动最复杂的地方在于：我希望在 *C-side* 控制服务的启动，所以我希望 *uv_loop_t* 在 *C-side* 中初始化和分配并 *uv_run*，但是 *luv* 这个库在封装中，是完全没有吧 *mainloop* 暴露出任何接口的，也无法直接在 *Lua-side* 初始化这个 *loop*

最后我找到的方法是：直接在 *C-side* 初始话服务的时候，初始化*lua_State*的时候，把*luv*这个库接在标准库后面直接加载，然后再加载前手动执行这个库中留好的一个 *luv_set_loop* 函数，把*lua_State*中*loop* 设置为我在 *C-side* 初始化分配好的。

这里我让*AI*帮我写了一个搜索*package.cpath*并手动加载*luv*的代码，非常舒适，技术改变生活！
